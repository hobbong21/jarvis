# Agent Design Patterns — 6 Architectures

Harness 가 도메인 분석 후 선택하는 여섯 가지 팀 아키텍처. 각 패턴에는 (a) 적합한
상황, (b) 비적합 상황, (c) 메시지 흐름, (d) 실패 모드와 그 처치가 정의된다.

## 1. Pipeline (파이프라인)

```
A ──▶ B ──▶ C ──▶ 출력
```

- 적합: 단계 간 강한 의존, 단방향 데이터 흐름.
- 비적합: 단계가 독립이거나, 중간 단계가 자주 실패해 재시도가 잦은 경우.
- 실패 처치: 각 단계는 자기 입력의 무결성만 검증. 실패는 상류로 짧게 보고.

## 2. Fan-out / Fan-in (팬아웃 / 팬인)

```
       ┌──▶ A ──┐
입력 ──┼──▶ B ──┼──▶ Aggregator ──▶ 출력
       └──▶ C ──┘
```

- 적합: 같은 입력에 대해 독립 분석을 병렬 수행 후 합병.
- 비적합: 분석 사이에 의존이 있거나 중복 비용이 큰 경우.
- 실패 처치: 일부 가지 실패 허용. Aggregator 가 부분 결과로 결정 가능해야 함.

## 3. Expert Pool (전문가 풀)

```
입력 ──▶ Router ──▶ (A | B | C 중 1)
```

- 적합: 상황(언어/비용/지연)에 따라 다른 전문가 호출.
- 비적합: 모든 입력에 동일 처리가 충분한 경우.
- 실패 처치: Router 는 폴백 우선순위 보유 (A 실패 → B → C).

## 4. Generate-Verify (생성-검증)

```
요구 ──▶ Generator ──▶ Verifier ──▶ (통과 ? 출력 : 재생성)
```

- 적합: 산출물 품질이 가변적이고 자동 검증이 가능한 경우.
- 비적합: 검증 신호가 약하거나 재생성 비용이 너무 큰 경우.
- 실패 처치: 재시도 상한과 명시적 "포기 시 사용자에게 보여줄 부분 결과" 정의.

## 5. Supervisor (감독자)

```
입력 ──▶ Supervisor ──▶ 동적 분배 (A, B, C, ...)
```

- 적합: 입력 형태가 다양해 정적 라우팅이 어려운 경우.
- 비적합: 입력이 단조롭고 라우팅 비용이 결정에 비해 큰 경우.
- 실패 처치: Supervisor 는 자기 결정 로그를 남겨 디버깅 가능해야 함.

## 6. Hierarchical Delegation (계층적 위임)

```
Lead ──▶ Sub-Lead ──▶ Worker
   └──▶ Sub-Lead ──▶ Worker
```

- 적합: 도메인이 크고 자연스러운 트리 구조를 가진 경우.
- 비적합: 모든 의사결정이 사실상 한 사람(에이전트) 에게 모이는 경우.
- 실패 처치: 각 레벨은 위 레벨에 요약만 보고. 디테일은 자기 트리에 보관.

## 결정 트리

1. 입력 → 출력이 **순차 단방향** 인가? → Pipeline.
2. 동일 입력에 대해 **병렬 독립 분석** 후 합쳐야 하는가? → Fan-out/Fan-in.
3. **상황에 따라 다른 전문가** 가 처리하는가? → Expert Pool.
4. 산출물에 **명시적 품질 게이트** 가 필요한가? → Generate-Verify.
5. 라우팅이 **사전에 정의 불가** 한가? → Supervisor.
6. 도메인이 **자연스러운 트리 구조** 인가? → Hierarchical.

복합 도메인은 패턴을 합성한다 (예: Supervisor + Expert Pool, Pipeline + Generate-
Verify 가 자주 등장).
